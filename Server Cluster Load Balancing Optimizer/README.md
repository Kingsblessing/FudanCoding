## 1 引言

随着大语言模型、AIGC 应用的普及，GPU 集群成为核心算力支撑，但实际运行中存在节点负载不均问题：部分节点高负载拥堵，部分节点低利用率闲置。任务迁移是解决该问题的关键，但迁移存在网络传输成本，且受链路带宽约束，因此需要设计兼顾负载均衡、成本优化与带宽限制的迁移方案。

本实验设计并实现了一款服务器集群负载均衡优化器，针对云平台 GPU 集群负载不均衡问题，通过任务迁移实现资源合理分配。优化器依次完成基础要求（满足容量约束的可行方案）、高级要求（最小化总迁移成本）和附加要求（考虑链路带宽约束的分轮次迁移），核心应用图结构表示、最短路径算法等数据结构与算法知识，最终实现负载均衡与约束满足的目标。

## 2 项目要求

- 基础要求：生成满足节点 GPU 容量约束的任务分配方案，不考虑迁移成本与带宽；
- 高级要求：在容量约束基础上，最小化总迁移成本；
- 附加要求：引入链路带宽约束，分轮次完成迁移，统计迁移轮次。

## 3 整体实现思路

### 3.1 数据结构设计

为适配集群、任务与网络的核心信息，设计三类核心结构体及辅助存储结构：

1. **节点（node）**：存储节点编号、最大 GPU 容量、已使用 GPU 数量，支持直接构造初始化；
2. **链路（edge）**：存储相连节点编号、单 GPU 迁移成本、带宽容量、剩余带宽，提供双向链路统一键值（`get_key`方法），便于快速查找；
3. **任务（task）**：存储任务编号、初始节点、GPU 需求、最终节点、迁移成本、迁移路径及迁移状态；
4. **辅助结构**：
   1. 最短路径成本矩阵`distant`与前驱节点矩阵`prenode`：存储所有节点对之间的最短路径成本与路径信息；
   2. 哈希表`edgemap`：通过链路键值快速索引对应的链路对象，降低带宽检查时的查找开销。

### 3.2 核心算法选择

- **最短路径计算**：采用 Floyd-Warshall 算法，因节点数量 N≤50，O(N³)时间复杂度可满足性能要求，且能一次性计算所有节点对之间的最短路径，适配高级要求中迁移成本计算的需求。

### 3.3 整体流程

1. 输入数据读取与初始化：解析节点、链路、任务信息，构造数据结构实例；
2. 最短路径预处理：通过 Floyd-Warshall 算法填充`distant`和`prenode`矩阵；
3. 分阶段执行要求：依次运行基础要求、高级要求、附加要求的迁移逻辑；
4. 结果输出：按各阶段格式要求输出任务分配、节点负载、成本/轮次等信息。

## 4 各模块实现思路

### 4.1 基础要求实现

目标：在不考虑迁移成本与带宽的前提下，满足节点容量约束并初步平衡负载。

1. 初始化重置所有节点的已使用 GPU 数量为0；
2. 任务分配：
   1. 遍历每个任务，筛选出“已用 GPU + 任务需求 ≤ 节点容量“的可行节点；
   2. 从可行节点中选择“已用 GPU 最少”的节点作为目标节点；
   3. 更新节点已用 GPU 数量，设置任务最终节点与迁移成本。
3. 输出任务分配结果与各节点最终负载。

### 4.2 高级要求实现

目标：在容量约束基础上，最小化总迁移成本。

#### 优化策略设计思路

采用“优先保留+成本最优”策略，平衡迁移成本与负载约束：

1. 优先保留原节点：若任务初始节点能容纳该任务（已用+需求≤容量），直接分配至原节点，迁移成本为0（避免无意义迁移）；
2. 成本最优选择：若原节点不可用，遍历所有可行节点，计算“原节点到目标节点的最短路径成本 × 任务GPU需求”（单任务迁移成本），选择成本最低的节点；
3. 路径回溯：通过前驱节点矩阵`prenode`回溯最短路径，反转后得到从初始节点到目标节点的迁移路径。

#### 关键实现步骤

1. 状态重置：恢复节点已用 GPU 数量与任务状态；
2. 按“原节点优先→成本最优”逻辑分配目标节点；
3. 计算单任务迁移成本，回溯并存储迁移路径；
4. 累加所有任务迁移成本，得到总迁移成本。

### 4.3 附加要求实现

目标：在高级要求的迁移方案基础上，满足链路带宽约束，分轮次完成迁移。

#### 设计思路

1. 带宽约束：每条链路每轮最多支持`bandwidth`个任务迁移；
2. 轮次迭代：
   1. 每轮开始时，重置所有链路的剩余带宽；
   2. 遍历未迁移任务，检查其迁移路径上所有链路的剩余带宽是否充足；
   3. 若带宽充足，占用链路带宽（剩余带宽 - 1），标记任务为已迁移，记录本轮迁移日志；
3. 终止条件：所有任务均完成迁移，迭代结束。

#### 输出格式

在高级要求输出基础上，新增两类关键信息：

1. 总迁移轮次：完成所有任务迁移所需的时间轮次（`count`变量）；
2. 迁移日志：按“轮次 任务编号 初始节点 最终节点”格式，输出每轮完成的迁移任务，清晰展示迁移时序。

#### 高级与附加要求对比分析

| 指标                | 高级要求 | 附加要求 | 差异原因                                                     |
| ------------------- | -------- | -------- | ------------------------------------------------------------ |
| 总迁移成本          | 15       | 15       | 附加要求未改变任务最终分配方案，仅调整迁移时序，成本保持一致 |
| 迁移轮次            | 1        | 3        | 链路带宽为1，任务2、5共享1→3链路，任务3占用1→2链路，需分3轮完成迁移 |
| 迁移效率（任务/轮） | 5        | 2/2/1    | 带宽约束限制了每轮迁移任务数量，路径重叠的任务需错峰迁移     |

#### 带宽约束影响分析

1. 对总迁移成本：无影响。带宽约束仅限制迁移的轮次，不改变任务的目标节点与迁移路径，因此总迁移成本与高级要求一致；
2. 对完成时间：显著延长。带宽越低、路径重叠度越高，轮次越多。例如当链路带宽为2时，上述样例迁移轮次可降至二轮，验证了带宽与迁移效率正相关。

## 6 收获与总结

本实验成功实现了服务器集群负载均衡优化器的三类核心要求，通过合理的数据结构设计与算法选择，兼顾了负载均衡、成本优化与带宽约束。

通过本次实验，我深入理解了 Floyd-Warshall 算法在最短路径计算中的优势，掌握通过前驱矩阵回溯路径的方法；通过哈希表`edgemap`优化链路查找效率，使用`emplace_back`避免默认构造函数，提升代码性能；学会分层次处理多约束问题（容量→成本→带宽），在满足基础约束的前提下逐步优化目标。

项目的核心价值在于将数据结构与算法知识应用于实际工程问题，实现了“理论到实践”的转化，同时为后续优化提供了基础框架。
